# /wos:audit-chain

**Goal:** Create `skills/audit-chain/SKILL.md` — a single skill with two invocation modes:
(1) **goal mode** — structured dialogue that produces a valid `*.chain.md` manifest from a
free-text workflow goal; (2) **manifest mode** — structural checks via `wos/chain.py`,
LLM cross-reference checks against the referenced SKILL.md bodies, per-finding repair
with per-change user confirmation, and a re-verify cycle until clean. Closes bcbeidel/wos#225.

**Scope:**

Must have:
- `skills/audit-chain/SKILL.md` with both invocation modes in a single skill
- Goal mode: discover available skills → structured dialogue (sequence → output contracts
  → input requirements → gates → termination + negative-scope) → produce
  `plans/YYYY-MM-DD-<name>.chain.md` → **hard gate** (design only, no skill execution)
- Manifest mode: structural checks (via `scripts/lint.py`) first → LLM cross-reference
  check per step → findings table → per-finding repair loop → re-verify → repeat until
  clean or user exits
- Repair loop handles gap case: missing skill → invoke `/wos:build-skill` inline
- Skill passes `scripts/lint.py` with no new failures or warnings
- Findings output format matches `scripts/lint.py` style (file, issue, severity)

Won't have:
- New Python validators or changes to `wos/chain.py` — structural checks are already
  implemented; this skill adds LLM-level cross-reference judgment on top
- Chain execution or dispatch — audit-chain designs and validates only, never runs the chain
- Automated test coverage for the skill (skill instructions are not Python code)
- Bulk repair without per-change confirmation

**Approach:** Pure SKILL.md authoring — no Python changes. The skill routes on argument
presence: no argument or free-text goal → goal mode; a `*.chain.md` path → manifest mode.
Manifest mode runs `scripts/lint.py` for deterministic structural checks first (fast fail),
then LLM cross-reference for semantic plausibility. The repair loop mirrors `audit-skill`:
present findings, prompt for selection, propose minimal edit, show diff, apply on
confirmation, re-verify. The hard gate in goal mode is a structural constraint in the
workflow — the skill must state explicitly that it does not invoke any step in the chain.

### Task 2: Write `skills/audit-chain/SKILL.md`

**Files:**
- Create: `skills/audit-chain/SKILL.md`

- [x] **Step 1:** Draft `skills/audit-chain/SKILL.md` with the following structure: <!-- sha:835c8a2 -->

  **Frontmatter:**
  - `name: audit-chain`
  - `description:` — first sentence: "Design a skill chain from a goal or audit an existing chain manifest for structural and contract issues." Second sentence: "Use when the user wants to 'design a chain', 'create a workflow', 'check a chain', 'audit a chain manifest', or 'repair a chain'." (`argument-hint: "[workflow goal or path/to/manifest.chain.md]"`, `user-invocable: true`)

  **Body — one-sentence summary:** State both modes up front so routing is unambiguous.

  **`## Workflow`** — two named branches:

  *Branch A — Goal mode (no manifest path argument, or free-text goal):*

  1. **Detect mode** — if argument is a path to a `*.chain.md` file, go to Branch B. Otherwise enter goal mode.
  2. **Discover skills** — list available skills from `skills/` (and any installed plugin `skills/` directories); present as a numbered list with descriptions so the user can select by name.
  3. **Structured dialogue** — collect in order:
     - Skill sequence (ordered list; user can reorder or add)
     - Output contract per skill (what artifact or state does this step produce?)
     - Input requirement per skill (what does this step need from the prior step?)
     - Gate condition per consequential step (when does a human review before continuing?)
     - Termination condition (what constitutes chain-complete success?)
     - Negative scope (what does this chain explicitly not do?)
  4. **Produce manifest** — write `plans/YYYY-MM-DD-<name>.chain.md` using the chain manifest format (frontmatter: `name`, `description`, `type: chain`, `goal`, `negative-scope`; body: `## Steps` pipe table with columns: Step | Skill | Input Contract | Output Contract | Gate).
  5. **Hard gate** — present the manifest for user review. Do not invoke any skill in the chain. State explicitly: "This is a design artifact only — invoke `/wos:execute-plan` or run each skill manually to execute."

  *Branch B — Manifest mode (argument is a `*.chain.md` path):*

  1. **Structural checks** — run `scripts/lint.py --root <project-root> --no-urls` and extract chain findings for the target manifest. These cover 5 structural dimensions: skills exist, contracts declared per step, gates on consequential steps, termination condition present, no cycles. Fast fail first — if structural failures exist, present them before cross-reference.
  2. **Cross-reference check** — for each step in the manifest: read the referenced SKILL.md body; assess whether the declared `output_contract` is plausible given what the skill actually produces. Flag mismatches as `warn` findings (not `fail`) — the user decides whether the contract or the SKILL.md is wrong.
  3. **Findings table** — present all findings (structural + cross-reference) in `scripts/lint.py` format:
     ```
     | File | Issue | Severity |
     |------|-------|----------|
     | plans/my.chain.md | Step 2 output contract 'summary.md' not described by distill skill | warn |
     ```
     Summary line: `N fail, N warn`. Sort: fail before warn; structural before cross-reference.
  4. **Repair loop** — after findings, ask: "Apply fixes? Enter y (all), n (skip), or comma-separated numbers." For each selected finding:
     - If the fix targets the manifest: propose a specific manifest edit, show diff, apply on confirmation.
     - If the fix targets a SKILL.md: propose a specific SKILL.md edit, show diff, apply on confirmation.
     - If the finding reveals a missing skill (step references a skill that doesn't exist): invoke `/wos:build-skill` inline to create it, then re-check.
  5. **Re-verify** — after each applied fix, re-run `scripts/lint.py` and re-run cross-reference on affected steps. Repeat from findings presentation.
  6. **Termination** — exit when all findings are resolved or the user declines all remaining fixes. If clean: "Chain manifest is well-formed — 0 issues found."

  **`## Anti-Pattern Guards`:**
  1. **Running cross-reference before structural checks** — `scripts/lint.py` structural checks are deterministic and fast; always run them first and surface failures before LLM judgment
  2. **Bulk repair without per-change confirmation** — per-change confirmation is required; applying all fixes at once removes the ability to review individual changes
  3. **Executing the chain during goal mode** — goal mode is design-only; the hard gate must be enforced; do not invoke any step of the chain
  4. **Treating cross-reference mismatches as failures** — cross-reference findings are `warn`, not `fail`; the user decides whether the contract or the SKILL.md needs updating
  5. **Crashing on malformed input** — partial or incomplete manifests must produce findings, not errors; if `parse_chain` raises, report it as a structural `fail` finding and continue

  **`## Handoff`:**
  - **Receives:** Free-text workflow goal (goal mode) or path to `*.chain.md` manifest (manifest mode)
  - **Produces:** Goal mode — `plans/YYYY-MM-DD-<name>.chain.md` manifest file. Manifest mode — structured findings table; optionally, targeted edits applied to the manifest or referenced SKILL.md files.
  - **Chainable to:** execute-plan (to run the chain steps), build-skill (for gap case in repair loop)

  **`## Key Instructions`:**
  - Goal mode is design-only — never invoke a skill in the chain; the manifest is the sole deliverable
  - Manifest mode cross-reference is LLM judgment — flag mismatches as warn only; let the user decide what to fix
  - Per-change confirmation is non-negotiable — never apply more than one fix without separate confirmation
  - `scripts/lint.py` structural checks are authoritative for the 5 structural dimensions; do not re-implement them in the skill
  - Won't: execute chain steps, bulk-apply fixes, or treat cross-reference mismatches as hard failures
